# Sanity: зачем нужен `company` и как его использовать

`company` нужен не как "настройки сайта", а как ключ изоляции данных внутри одного общего Sanity dataset.

Идея простая:

- dataset один
- компаний внутри него может быть несколько
- сайт сначала определяет, какая компания сейчас открыта
- потом все данные запрашиваются только для этой компании

## Зачем нужен `company`

Если в одном dataset лежат данные сразу нескольких компаний, без явного разделителя данные начнут смешиваться:

- информация одной компании могут попасть на сайт другой
- локальная разработка станет непредсказуемой
- невозможно будет безопасно масштабировать проект на несколько витрин

Документ `company` решает эту проблему. Он становится "корнем принадлежности" для всех company-scoped сущностей.

То есть логика такая:

1. Определяем текущую компанию по домену.
2. Получаем её `_id`.
3. Все запросы в Sanity фильтруем по `company._ref == $companyId`.

Поэтому `company` нужен не ради UI или SEO, а ради tenancy boundary: это граница, которая отделяет данные одной компании от другой.

## Как `company` устроен

У документа есть такие поля:

- `name` - человекочитаемое имя компании
- `slug` - стабильный идентификатор
- `primaryDomain` - основной домен компании
- `domains` - все hostname, которые должны мапиться на эту компанию
- `isDefault` - fallback для локальной разработки

Пример:

```json
{
  "_type": "company",
  "name": "Acme Store",
  "slug": {
    "_type": "slug",
    "current": "acme-store"
  },
  "primaryDomain": "acme.example.com",
  "domains": ["acme.example.com", "www.acme.example.com"],
  "isDefault": false
}
```

## Как `company` используется

Что происходит:

1. Из request headers читается `x-forwarded-host` или `host`.
2. Hostname нормализуется.
3. Если это `localhost` или `127.0.0.1`, берётся компания с `isDefault == true`.
4. Если это реальный домен, ищется `company`, где:

```groq
primaryDomain == $hostname || $hostname in domains
```

5. После этого все запросы к данным идут только через найденный `companyId`.

Именно поэтому `company` в этой архитектуре является обязательным верхнеуровневым reference для всех сущностей, которые должны быть разделены между компаниями.

## Как это выглядит на уровне запросов

Сначала выбирается компания:

```groq
*[
  _type == "company" &&
  (
    primaryDomain == $hostname ||
    $hostname in domains
  )
][0]
```

Потом любой company-scoped контент должен фильтроваться так:

```groq
*[
  _type == "someDocumentType" &&
  company._ref == $companyId
]
```

Это главный контракт архитектуры.

Неважно, как называются остальные документы в другом проекте. Если они должны принадлежать конкретной компании, у них должен быть reference на `company`, и запросы к ним должны фильтроваться по `company._ref`.

## Как правильно использовать `company` в другом проекте

Если у другого проекта другая предметная модель, сам подход всё равно остаётся тем же:

1. Завести отдельный документ `company`.
2. Хранить в нём домены, по которым определяется текущая компания.
3. Во все документы, которые должны быть изолированы по компании, добавить поле:

```ts
defineField({
  name: "company",
  title: "Company",
  type: "reference",
  to: [{ type: "company" }],
  validation: (rule) => rule.required(),
});
```

4. Любые выборки делать только после определения текущей компании.
5. Во всех GROQ-запросах фильтровать данные по `company._ref == $companyId`.

## Когда `company` особенно нужен

Такой слой нужен, если выполняется хотя бы одно условие:

- один Sanity dataset обслуживает несколько брендов или клиентов
- один frontend должен показывать разные данные в зависимости от домена
- нужно избежать смешивания контента между компаниями
- проект потенциально вырастет в multi-tenant схему

Если всего одна компания и это никогда не изменится, `company` может быть лишним. Но если multi-company или multi-brand хотя бы возможен, такой документ лучше закладывать сразу.

## Что важно донести другому разработчику

Главная мысль:

`company` здесь не про настройки сайта. Это системная сущность, которая определяет владельца данных.

То есть:

- `company` выбирается по домену
- `company._id` становится фильтром для данных
- все company-scoped документы обязаны ссылаться на `company`
- без этого изоляция данных не работает

Короткая формулировка для передачи:

"Мы используем документ `company` как tenant boundary внутри одного Sanity dataset. Сайт сначала определяет компанию по hostname, а потом читает только те документы, у которых `company._ref` совпадает с найденной компанией."
